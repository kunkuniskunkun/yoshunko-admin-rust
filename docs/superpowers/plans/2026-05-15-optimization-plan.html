🎮 Yoshunko Admin 优化方案

运行时性能 · 代码架构 · 发布与社区 — 渐进式优化路线图

📅 2026-05-15 📌 当前版本 v0.620-r 🎯 目标：发布到 GitHub 与玩家社区

一、现状分析

⚡

运行时性能

  • DataManager 缓存无失效机制，切换玩家后旧数据残留
  • get_avatars/get_weapons/get_equips 每次全量读取+解析所有 ZON 文件
  • 前端列表无虚拟滚动，角色多时 DOM 节点过多
  • 面板切换无骨架屏过渡，加载时白屏
  • 模板数据 ~50KB 一次性全量传输
  • 前端 cacheDirty 标记粗糙，无法精确刷新
🏗️

代码架构

  • theme.css 1700+ 行单文件，变量/组件/动画混杂
  • useAppState.ts 导出 30+ ref，职责不清
  • 9 个面板的 gallery/editor 切换逻辑高度重复
  • api.rs 830 行单文件，所有命令挤在一起
  • 错误处理全用 String，无类型化错误
  • 前端 api.ts launchProgram 参数名与后端不一致
🚀

发布与社区

  • 无 GitHub Actions CI/CD
  • 无自动更新机制（Tauri updater 未集成）
  • 无 Release Notes 模板/自动生成
  • CHANGELOG 完全手动维护
  • 无安装包签名（Windows SmartScreen 会警告）
  • build-run.ps1 硬编码路径，不可移植

二、实施策略选择

方案 A：渐进式优化 ✅ 推荐

按用户体验影响排序，分 4 个阶段逐步改进。每阶段独立可验证，不会引入大面积回归。优先解决用户可感知的问题，再做内部架构清理。

✅ 优势
  • 每步可独立验证，风险低
  • 随时可停，不影响现有功能
  • 用户尽早感受到改善
  • 适合个人项目节奏
⚠️ 劣势
  • 总工期略长（4 个迭代）
  • 中间状态可能不够"干净"
  • 需要更多版本管理

方案 B：集中式重构

一次性解决所有架构问题，建立完善基础设施后发布。改动范围大但一步到位。

✅ 优势
  • 一步到位，代码更干净
  • 不需要中间版本
⚠️ 劣势
  • 风险高，大范围改动易引入回归
  • 无自动化测试，验证困难
  • 期间无法发布中间版本
  • 对小白不友好
推荐方案 A：项目无自动化测试，集中重构的回归风险太高。渐进式优化可以每步验证，且用户体验改善最快。

三、渐进式优化路线图

Phase 1体验感知
Phase 2性能提升
Phase 3架构清理
Phase 4发布就绪
1
用户体验感知优化
高优先
  • 面板切换加载态：切换面板时显示骨架屏，消除白屏感 Vue UX 高感知
  • 保存操作反馈：保存时按钮显示 loading 状态 + 成功/失败 toast Vue UX 高感知
  • 缓存精确刷新：切换玩家时清空旧缓存，避免显示上一个玩家的数据 Rust Vue 高感知
  • 错误提示优化：ZON 解析失败时前端显示友好提示，而非静默失败 Rust Vue 中感知
  • 未保存提醒：dirty 状态下切换面板/关闭窗口时弹出确认 Vue UX 高感知
2
运行时性能提升
中优先
  • DataManager 缓存失效：切换玩家时清除缓存，写入后标记脏数据 Rust 高影响
  • 列表接口优化：get_avatars 等列表接口只读文件名+首行元数据，不解析完整 ZON Rust 高影响
  • 前端虚拟列表：角色/武器/驱动盘超过 50 项时启用虚拟滚动 Vue 中影响
  • 模板数据懒加载：templates 按需分片传输，首次只加载名称/稀有度 Rust Vue 低影响
  • 并行文件读取：列表接口用 rayon 并行读取 ZON 文件 Rust 中影响
3
代码架构清理
低优先
  • theme.css 拆分：按组件拆为 variables.css / base.css / sidebar.css / cards.css / forms.css / animations.css CSS 中影响
  • useAppState 拆分：按职责拆为 usePlayer.ts / useAvatar.ts / useWeapon.ts / useEquip.ts / useEditor.ts Vue 中影响
  • 面板逻辑复用：抽取 useGalleryEditor composable，统一 gallery/editor 切换模式 Vue 中影响
  • api.rs 拆分：按领域拆为 config.rs / players.rs / avatars.rs / weapons.rs / equips.rs / hadal.rs / launch.rs Rust 低影响
  • 错误类型化：定义 AppError 枚举替代 String，前端可按类型处理 Rust 低影响
4
发布就绪
高优先
  • GitHub Actions CI：push tag 时自动构建 + 生成 Release + 上传 NSIS 安装包 Infra 高影响
  • Tauri Updater 集成：应用内自动检查更新 + 增量更新 Rust Infra 高影响
  • CHANGELOG 自动化：从 git commit 生成 CHANGELOG，配合 conventional commits Infra 中影响
  • Release Notes 模板：GitHub Release 自动填充版本号/变更/截图 Infra 中影响
  • build-run.ps1 通用化：移除硬编码路径，支持任意开发环境 Infra 低影响

六、风险与缓解

风险影响缓解措施
无自动化测试，重构引入回归高每步手动验证核心流程；优先做用户可感知的改动
CSS 拆分后样式丢失中拆分后立即全量视觉检查；@import 由 Vite 合并，无运行时差异
虚拟滚动与 Naive UI 冲突中先在驱动盘面板（最多项）试点；保留降级路径
GitHub Actions 构建环境差异低先本地构建验证，再迁移到 CI
Tauri Updater 需要签名中Phase 4 再处理；无签名也可发布，只是 SmartScreen 警告
